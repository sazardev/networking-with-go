# Common Network Protocols (HTTP, FTP, DNS, etc.)

> "Imagine a bustling city where everyone speaks different languages, but thanks to interpreters (protocols), everyone can communicate and get things done!"

---

## What Is a Network Protocol?

A network protocol is a precisely written agreement about three things: **syntax** (what the bytes look like on the wire), **semantics** (what each message means), and **timing/state** (what order messages must arrive in, and what to do if something doesn't show up). Without that agreement, two computers exchanging bytes are just noise to each other — a browser sending raw bytes to a mail server produces nothing but a confused disconnection, because neither side agrees on what the bytes mean.

Almost every protocol you'll meet in this book started life as an RFC (Request for Comments) — a public, versioned specification maintained mostly by the IETF. That "public spec" model is why the internet works at all: a Go program you write today can talk HTTP to a web server running on a completely different operating system, written in a completely different language, twenty years from now, because both sides implement the same publicly documented rules rather than some vendor's private convention. This is also why protocols are so slow and careful to change — HTTP/1.1 is still everywhere almost thirty years after HTTP/1.0, because breaking the agreement breaks the internet for everyone who depends on it.

## The All-Stars of Networking

### 1. HTTP/HTTPS (HyperText Transfer Protocol / Secure)
- **Analogy:** Like a waiter taking your order and bringing food (web pages, JSON, images) to your table (browser or app).
- **How it works:** A client sends a request line (`GET /path HTTP/1.1`), a set of headers, and optionally a body; the server replies with a status line (`HTTP/1.1 200 OK`), headers, and a body. Methods like `GET`, `POST`, `PUT`, `DELETE` describe intent; status codes like `200`, `404`, `500` describe the outcome.
- **Use Case:** Browsing websites, calling REST/JSON APIs, streaming manifests.
- **Versions matter:** HTTP/1.1 opens one request at a time per connection (or pipelines awkwardly); HTTP/2 multiplexes many requests over one TCP connection using binary framing; HTTP/3 goes further and runs over QUIC/UDP instead of TCP (see Chapter 08's deep dive on why).
- **Fun Fact:** HTTPS is just HTTP layered on top of TLS — same requests and responses, wrapped in an encrypted, authenticated tunnel. See Chapter 10 for how that encryption is actually established.

```
[Browser] <---HTTP/HTTPS---> [Web Server]
```

### 2. FTP (File Transfer Protocol)
- **Analogy:** Like a moving truck, helping you send big boxes (files) between houses (computers).
- **How it works:** FTP is unusual in that it uses **two** TCP connections — a control connection (port 21) for commands like `USER`, `PASS`, `RETR`, `STOR`, and a separate data connection for the actual file bytes. In **active mode** the server connects back to the client to send data; in **passive mode** the client opens both connections itself. Active mode routinely breaks behind NAT and firewalls (the server trying to connect back inward looks like an unsolicited inbound connection), which is exactly why passive mode became the default almost everywhere — a nice preview of the NAT/firewall issues covered in Chapter 11.
- **Use Case:** Uploading/downloading files to servers, legacy publishing workflows.
- **Fun Fact:** FTP predates email — it was standardized in 1971. Plain FTP sends credentials and file contents unencrypted, which is why SFTP (file transfer over SSH) and FTPS (FTP wrapped in TLS) replaced it almost everywhere credentials matter.

```
[Client] <---FTP (control + data)---> [Server]
```

### 3. DNS (Domain Name System)
- **Analogy:** Like a phone book for the internet — translates names (`www.example.com`) into numbers (IP addresses).
- **How it works:** DNS is a distributed hierarchy, not one giant database. Your resolver asks a **root** server "who handles `.com`?", then asks that **TLD** server "who handles `example.com`?", then asks that domain's **authoritative** server for the actual `A`/`AAAA` record. In practice, most of that walk is skipped because your ISP or a public resolver (like `8.8.8.8` or `1.1.1.1`) caches answers for the record's **TTL** (time-to-live), so repeat lookups are nearly instant.
- **Common record types:** `A` (IPv4 address), `AAAA` (IPv6 address), `CNAME` (alias to another name), `MX` (mail server), `TXT` (arbitrary text, often used for domain verification).
- **Use Case:** Every time you visit a website, connect to an API, or send an email.
- **Fun Fact:** The first DNS was deployed in 1983 — before that, every machine on the early internet shared a single, manually updated `HOSTS.TXT` file.

```
[You] ---(www.example.com)---> [DNS Resolver] ---(93.184.216.34)---> [Website]
```

### 4. SMTP, POP3, IMAP (Email Protocols)
- **Analogy:** Like the postal service — SMTP sends mail, POP3/IMAP help you receive and organize it.
- **How they differ:** SMTP (port 25 between servers, 587 for client submission) is strictly for *sending*. POP3 (port 110) downloads messages to one device and typically deletes them from the server — fine in the 1990s when you only had one computer, awkward now that you check mail from three devices. IMAP (port 143) keeps mail on the server and *synchronizes* state (read/unread, folders) across every device, which is why virtually every modern mail app defaults to IMAP.
- **Fun Fact:** The `@` symbol in email addresses was chosen by Ray Tomlinson in 1971 specifically because it almost never appeared in a person's name, so it made an unambiguous separator between user and host.

```
[Sender] --SMTP--> [Mail Server] --IMAP/POP3--> [Recipient]
```

### 5. DHCP (Dynamic Host Configuration Protocol)
- **Analogy:** Like a hotel receptionist assigning room numbers (IP addresses) to guests (devices) as they check in.
- **How it works:** DHCP runs a four-step exchange nicknamed **DORA**: the device broadcasts **D**iscover ("is anyone a DHCP server?"), a server responds with an **O**ffer (a candidate IP), the device broadcasts a **R**equest confirming it wants that offer, and the server replies with an **A**ck, handing out the address along with the subnet mask, default gateway, and DNS servers to use.
- **Use Case:** Automatically configuring every device that joins a network, without a human typing in IP addresses by hand.
- **Fun Fact:** Without DHCP, every phone, laptop, and IoT device joining a coffee shop's Wi-Fi would need someone to manually assign it a unique address — DHCP is the reason "just connect to Wi-Fi" actually works.

### 6. SSH (Secure Shell)
- **Analogy:** Like a secret, authenticated tunnel into a castle — lets you control a computer remotely, securely.
- **How it works:** SSH (port 22) negotiates an encrypted channel, then authenticates you (usually via a public/private key pair rather than a password), and finally gives you a remote shell, a file-transfer channel (SFTP/SCP), or even an arbitrary tunneled TCP connection through `-L`/`-R` port forwarding.
- **Use Case:** Remote server administration, secure file transfer, tunneling other protocols.
- **Fun Fact:** SSH replaced Telnet specifically because Telnet sent every keystroke — including passwords — as plain, readable text.

### 7. Other Notable Protocols
- **Telnet (port 23):** Old remote login protocol; still occasionally used for talking to unencrypted device consoles, but never for anything sensitive.
- **SNMP (ports 161/162):** Lets network equipment report health and statistics to a monitoring system, and lets that system push configuration changes.
- **NTP (port 123):** Keeps clocks in sync across the internet — critical for TLS certificate validation, log correlation, and distributed systems that reason about "before" and "after."
- **LDAP (port 389, or 636 for LDAPS):** A directory-lookup protocol, effectively a company phonebook for usernames, groups, and permissions.

## Protocol Cheat Sheet

| Protocol | Default Port | Transport | Primary Use |
|---|---|---|---|
| HTTP | 80 | TCP | Unencrypted web traffic |
| HTTPS | 443 | TCP (TLS), or UDP via QUIC for HTTP/3 | Encrypted web traffic |
| FTP | 21 (control) / 20 (active data) | TCP | File transfer |
| SFTP | 22 | TCP (over SSH) | Secure file transfer |
| DNS | 53 | UDP (falls back to TCP for large replies) | Name resolution |
| SMTP | 25 / 587 | TCP | Sending email |
| POP3 | 110 | TCP | Download-and-delete email |
| IMAP | 143 | TCP | Synchronized email |
| DHCP | 67 / 68 | UDP | Automatic IP configuration |
| SSH | 22 | TCP | Secure remote shell / tunneling |
| Telnet | 23 | TCP | Insecure remote shell (legacy) |
| SNMP | 161 / 162 | UDP | Device monitoring |
| NTP | 123 | UDP | Clock synchronization |
| LDAP | 389 (636 for LDAPS) | TCP | Directory services |

<Warning title="Don't assume a protocol name implies a transport">It's tempting to assume "DNS is UDP" and stop there, but DNS explicitly falls back to TCP whenever a response is too large for one UDP datagram or a zone transfer is requested — the protocol was designed with both transports in mind from the start. Likewise, "HTTPS is TCP" was true for decades but is no longer universal now that HTTP/3 runs over UDP via QUIC. Always check the specific version and context rather than treating a protocol-to-transport mapping as fixed forever.</Warning>

<DeepDive title="Recursive vs. iterative DNS resolution">When your laptop asks its configured DNS resolver (often your ISP's, or a public one like `1.1.1.1`) for an address, it makes one **recursive** query and expects a final answer — the resolver does all the work of walking the hierarchy on your behalf. But the resolver itself talks to the root, TLD, and authoritative servers using **iterative** queries: each of those servers just replies "I don't know the final answer, but here's who to ask next," and the resolver follows that chain itself. This split matters in practice because it's *why* DNS scales: your device only ever makes one lookup, while the heavy hierarchical walking (and the caching that avoids repeating it) happens inside infrastructure you never talk to directly.</DeepDive>

## Protocols in Action: A Web Page Load

1. **DNS:** Your browser asks a resolver for the website's IP address (often already cached from a TTL-bounded earlier lookup).
2. **TCP handshake:** Your OS opens a TCP connection to that IP on port 443.
3. **TLS handshake:** Certificates are exchanged and verified, and a shared encryption key is negotiated (Chapter 10 covers this step in depth).
4. **HTTP/HTTPS:** Your browser sends the actual `GET` request over the now-encrypted connection and receives the page.
5. **More DNS + HTTP:** The HTML references images, scripts, and fonts on other hosts, triggering more DNS lookups and more HTTP requests, often reusing connections wherever possible.

<Axiom>A protocol is not the wire format alone — it's the agreed sequence of who speaks when, and what silence or a wrong answer is supposed to mean.</Axiom>

## Try It Yourself in Go

One short program can exercise two completely different protocols: DNS (via the standard library's resolver) and raw HTTP/1.0 (by hand-writing the request over a plain TCP socket, the same way `net/http` does it internally, just without the convenience wrapper).

```go
package main

import (
	"bufio"
	"fmt"
	"net"
	"time"
)

func main() {
	fmt.Println("--- DNS: resolving a hostname ---")
	lookupDemo("example.com")

	fmt.Println()
	fmt.Println("--- HTTP/1.0: a raw request over a TCP socket ---")
	httpDemo("example.com:80", "example.com")
}

func lookupDemo(host string) {
	addrs, err := net.LookupHost(host)
	if err != nil {
		fmt.Println("lookup error:", err)
		return
	}
	fmt.Printf("%s resolves to: %v\n", host, addrs)
}

func httpDemo(addr, hostHeader string) {
	conn, err := net.DialTimeout("tcp", addr, 5*time.Second)
	if err != nil {
		fmt.Println("dial error:", err)
		return
	}
	defer conn.Close()

	conn.SetDeadline(time.Now().Add(5 * time.Second))

	reqFmt := "GET / HTTP/1.0\r\nHost: %s\r\nConnection: close\r\n\r\n"
	request := fmt.Sprintf(reqFmt, hostHeader)
	if _, err := conn.Write([]byte(request)); err != nil {
		fmt.Println("write error:", err)
		return
	}

	scanner := bufio.NewScanner(conn)
	lineCount := 0
	for scanner.Scan() && lineCount < 5 {
		fmt.Println(scanner.Text())
		lineCount++
	}
}
```

Running this prints the resolved IPv4/IPv6 addresses for `example.com`, followed by the first handful of lines of a raw HTTP response — typically the status line (`HTTP/1.0 200 OK` or similar), then headers like `Content-Type` and `Date`. It demonstrates that `net.LookupHost` is DNS in action (the "translate a name to an address" job from earlier in this chapter), while `httpDemo` is HTTP in action — no `net/http` package involved at all, just a TCP socket and a manually written request line and headers, which is exactly what HTTP *is* underneath every framework. Note `Connection: close`, which tells the server not to keep the connection open waiting for a follow-up request, so the response finishes cleanly and our scanner can read it.

## Fun Facts & Memes
- If DNS fails, it's like forgetting everyone's phone number — no websites for you, even though the servers themselves are running fine.
- FTP is older than email.
- If protocols were people: HTTP is a chatty waiter, DNS is the phonebook, and SSH is the secret agent who only speaks in ciphers.
