# TCP vs UDP: Concepts and Use Cases

> "Imagine sending a package: with TCP, you get tracking, delivery confirmation, and a signature on arrival. With UDP, you toss the package over the fence and hope for the best!"

---

## What Are TCP and UDP, Really?

TCP (Transmission Control Protocol) and UDP (User Datagram Protocol) both live at the **transport layer** — the layer directly above IP that is responsible for getting data from one *process* on one machine to another *process* on another machine, using ports to tell processes apart. IP alone only promises "I'll try to get this packet from host A to host B." It says nothing about whether the packet arrives, arrives once, or arrives in order. TCP and UDP are the two answers the internet gives to the question: "given that IP is unreliable, what guarantees — if any — do you want on top of it?"

TCP answers with "all of them, and I'll manage the bookkeeping for you." UDP answers with "none, do it yourself if you need to." Neither answer is wrong — they are optimized for different jobs, and a huge amount of real-world networking skill is knowing which one a given problem actually needs.

---

## TCP: The Reliable Courier

**Analogy:** TCP is like a certified mail service. Every package (data packet) is tracked, delivered in order, and signed for. If something goes missing, it's resent until it arrives safely.

- **Features:**
  - Reliable, ordered delivery
  - Error checking and correction (checksums plus retransmission)
  - Connection-oriented (a handshake happens before any application data flows)
  - Flow control (the receiver tells the sender how much it can buffer)
  - Congestion control (the sender adapts to how congested the *network*, not just the receiver, is)
  - Higher overhead, but trustworthy

- **Examples:** Web browsing (HTTP/HTTPS), email (SMTP, IMAP, POP3), file transfers (FTP, SFTP), database connections, SSH sessions.

- **Diagram (three-way handshake, then data, then acknowledgments):**

```
[Sender] ---(SYN)-------> [Receiver]
[Sender] <--(SYN-ACK)---- [Receiver]
[Sender] ---(ACK)-------> [Receiver]
[Sender] ---(Data 1)----> [Receiver]
[Sender] ---(Data 2)----> [Receiver]
[Receiver] <--(ACK 1)---- [Sender]
[Receiver] <--(ACK 2)---- [Sender]
```

<DeepDive title="Why three messages, not two?">The handshake is SYN, SYN-ACK, ACK because both sides need to prove they can both send and receive, and both need to agree on a starting sequence number for their half of the conversation. Two messages would only prove one direction works. There is also a subtler reason: TCP has no way to know if a SYN it receives is from a brand-new connection attempt or a stray duplicate from an old, already-closed connection that got delayed in the network. The three-way exchange, combined with random initial sequence numbers, makes it extremely unlikely that a stale packet from a previous connection gets accidentally accepted as part of a new one. Closing a TCP connection is its own small dance too — usually a four-step FIN/ACK exchange, or an abrupt RST when one side just wants to bail out immediately (for example, when a server is overloaded and rejects a connection outright).</DeepDive>

TCP also does **flow control** (so a fast sender doesn't overwhelm a slow receiver's buffer) and **congestion control** (so a sender backs off when the *network path* itself, not just the receiver, seems overloaded — this is what algorithms like slow start, Reno, Cubic, and BBR are for). Both mechanisms are invisible to the application: you call `Write` and `Read`, and the kernel handles pacing, retransmission, and reordering for you.

<Warning title="TCP does not preserve your message boundaries">A very common bug for people new to sockets: TCP is a **byte stream**, not a sequence of messages. If you call `Write([]byte("hello"))` and then `Write([]byte("world"))`, the receiver might see one `Read` return `"helloworld"`, or it might see `"hel"` followed by `"loworld"` — the network and OS are free to coalesce or split your writes however they like. If your protocol needs distinct messages, you must build that framing yourself, typically with a length prefix or a delimiter, and read exactly that much. This is different from UDP, where each `Write` (or `WriteTo`) becomes exactly one datagram on the wire and the receiver's `Read` gets that whole datagram in one call (or nothing).</Warning>

## UDP: The Speedy Sprinter

**Analogy:** UDP is like sending postcards — fast, no tracking, and no guarantee they'll arrive. Great for when speed matters more than reliability.

- **Features:**
  - Unreliable, unordered delivery (packets can arrive out of order, duplicated, or not at all)
  - No error correction or retransmission built in
  - Connectionless — there is no handshake, no negotiated state, just "send this datagram"
  - Message-boundary preserving — each send is one discrete datagram
  - Tiny header (8 bytes, versus TCP's 20+), so lower overhead
  - Supports broadcast and multicast, which connection-oriented TCP cannot do

- **Examples:** Live video/audio streaming, online gaming, VoIP, DNS lookups, NTP time sync, DHCP.

- **Diagram:**

```
[Sender] ---(Data 1)---> [Receiver]
[Sender] ---(Data 2)---> [Receiver]
(No handshake, no ACKs, no guarantees!)
```

UDP is not "TCP but broken" — it is TCP with the reliability machinery deliberately removed so the application can decide what it actually needs. A live video call doesn't want a lost frame from three seconds ago retransmitted; by the time it arrives it's useless, and worse, TCP would stall *newer* frames behind it while waiting for the retransmission to complete. That stall is called **head-of-line blocking**, and it's one of the biggest reasons real-time and latency-sensitive systems reach for UDP instead of TCP.

<Axiom>TCP buys reliability with latency and lets the kernel manage that trade-off for you; UDP buys speed by handing that same decision back to your application.</Axiom>

## TCP vs UDP: Head-to-Head

| Dimension | TCP | UDP |
|---|---|---|
| Connection model | Connection-oriented (handshake, teardown) | Connectionless |
| Reliability | Guaranteed delivery, retransmits lost data | Best-effort, no retransmission |
| Ordering | Guaranteed in-order delivery | No ordering guarantee |
| Message boundaries | None — byte stream | Preserved — one send = one datagram |
| Flow control | Yes (receiver-driven window) | None |
| Congestion control | Yes (sender adapts to network load) | None (app must implement if needed) |
| Header overhead | 20+ bytes | 8 bytes |
| Multicast/broadcast | Not supported | Supported |
| Typical latency profile | Higher, especially under packet loss | Lower, consistent |
| Typical use cases | Web, email, file transfer, databases, SSH | Streaming, gaming, VoIP, DNS, NTP |

## Real-World Scenarios

1. **Online banking (TCP):** Every byte of your login and transaction request must arrive intact, in order, exactly once. Losing or duplicating a byte here is unacceptable, so the cost of TCP's overhead is worth paying.
2. **Video conferencing, e.g. Zoom or Google Meet (UDP, usually):** The audio/video media itself typically rides on RTP over UDP, because a dropped frame from half a second ago is worthless once new frames exist. Interestingly, when a corporate firewall blocks UDP outright, these same apps fall back to relaying media over TCP (via a TURN server) — it works, but call quality usually gets noticeably worse under packet loss, which is a very visible demonstration of head-of-line blocking in the real world.
3. **DNS lookups (UDP, with a TCP fallback):** A DNS query is normally a single small UDP datagram — fast, connectionless, cheap. But if the answer is too large to fit in one UDP datagram, the server sets the "truncated" flag and the resolver retries the same query over TCP, because now ordering and reliability for a multi-packet response actually matter.
4. **Competitive online games, e.g. Counter-Strike or Fortnite (UDP):** Player position updates arrive dozens of times per second. If one is lost, the next one arriving a few milliseconds later already supersedes it — there's no point retransmitting stale position data.
5. **File downloads and package managers, e.g. `apt`, `npm install` (TCP):** Every byte of a `.zip` or binary must be correct, or the archive fails to open. Speed matters, but correctness matters more.

<DeepDive title="Why HTTP/3 moved from TCP to UDP">For decades "the web runs on TCP" was close to gospel, since HTTP/1.1 and HTTP/2 both use it. HTTP/3 broke that pattern by building on QUIC, which runs over UDP. The reason is head-of-line blocking again, but at a different layer: HTTP/2 multiplexes many requests over a single TCP connection, so if even one TCP segment is lost, *every* multiplexed stream stalls until it's retransmitted — even requests that had nothing to do with the lost data. QUIC re-implements reliability, ordering, and congestion control itself, but does it **per stream** on top of UDP, so a lost packet only blocks the one stream it belonged to. This is a good example of a general pattern: UDP is often chosen not because an application wants *no* reliability, but because it wants to build its *own*, smarter version of reliability instead of accepting TCP's one-size-fits-all guarantees.</DeepDive>

## Try It Yourself in Go

The best way to feel the difference between "connection-oriented stream" and "connectionless datagram" is to run a tiny echo exchange over each and watch how little ceremony UDP needs compared to TCP.

```go
package main

import (
	"fmt"
	"net"
	"time"
)

func main() {
	fmt.Println("--- TCP demo ---")
	tcpDemo()

	fmt.Println("--- UDP demo ---")
	udpDemo()
}

func tcpDemo() {
	ln, err := net.Listen("tcp", "127.0.0.1:0")
	if err != nil {
		fmt.Println("listen error:", err)
		return
	}
	defer ln.Close()

	go func() {
		conn, err := ln.Accept() // completes the three-way handshake
		if err != nil {
			return
		}
		defer conn.Close()
		buf := make([]byte, 64)
		n, _ := conn.Read(buf)
		conn.Write(buf[:n]) // echo it straight back
	}()

	conn, err := net.DialTimeout("tcp", ln.Addr().String(), 2*time.Second)
	if err != nil {
		fmt.Println("dial error:", err)
		return
	}
	defer conn.Close()

	conn.Write([]byte("hello over TCP"))
	buf := make([]byte, 64)
	n, _ := conn.Read(buf)
	fmt.Printf("TCP echoed: %q\n", buf[:n])
}

func udpDemo() {
	// No handshake here, just a bound socket.
	pc, err := net.ListenPacket("udp", "127.0.0.1:0")
	if err != nil {
		fmt.Println("listen error:", err)
		return
	}
	defer pc.Close()

	go func() {
		buf := make([]byte, 64)
		n, addr, err := pc.ReadFrom(buf)
		if err != nil {
			return
		}
		pc.WriteTo(buf[:n], addr) // echo the datagram back to its sender
	}()

	conn, err := net.Dial("udp", pc.LocalAddr().String())
	if err != nil {
		fmt.Println("dial error:", err)
		return
	}
	defer conn.Close()

	conn.SetDeadline(time.Now().Add(2 * time.Second))
	conn.Write([]byte("hello over UDP"))
	buf := make([]byte, 64)
	n, err := conn.Read(buf)
	if err != nil {
		fmt.Println("read error:", err)
		return
	}
	fmt.Printf("UDP echoed: %q\n", buf[:n])
}
```

Run this with `go run main.go` and you'll see both `"TCP echoed: ..."` and `"UDP echoed: ..."` printed. Functionally the two demos look almost identical from the application's point of view — write some bytes, read some bytes back — which is exactly the point: Go's `net` package deliberately hides the plumbing differences behind a similar `Read`/`Write` interface. What's different is everything happening underneath. The TCP side quietly ran a SYN/SYN-ACK/ACK handshake before `Accept` even returned, and it's using a `net.Conn` bound to one specific peer for the life of that connection. The UDP side skipped all of that: `net.ListenPacket` just opened a socket ready to receive datagrams from *anyone*, and `ReadFrom`/`WriteTo` return the sender's address explicitly, because with UDP there is no persistent "connection" recording who you're talking to — every datagram carries its own destination.

## Fun Facts & Memes
- TCP's three-way handshake is like a secret handshake before a club meeting.
- UDP is sometimes called the "Unreliable Data Protocol" — but it's perfect when you need speed and can tolerate loss.
- If TCP were a person, it'd be a careful librarian; UDP would be a courier who's already sprinting to the next delivery before you've finished signing for the last one.
