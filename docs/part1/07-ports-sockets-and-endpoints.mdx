# Ports, Sockets, and Endpoints

> "Imagine a massive apartment building (the internet), where each apartment (IP address) has hundreds of doors (ports). Sockets are the keys that open these doors, letting messages in and out!"

---

## What is a Port?

A port is like a numbered door on your device. While your IP address is your building's street address, ports are the specific doors where data can enter or leave. Ports exist because a single IP address usually runs many programs at once—your laptop might simultaneously be browsing a website, checking email, and syncing a file backup, all while other devices on the internet only know your one IP address. Ports are how the operating system decides which of those several running programs a particular arriving packet belongs to.

- **Analogy:** Think of a hotel—one address, but many rooms. Each room (port) hosts a different guest (service).
- **Examples:**
  - Port 80: Web traffic (HTTP)
  - Port 443: Secure web traffic (HTTPS)
  - Port 25: Email (SMTP)
  - Port 53: DNS

- **Diagram:**

```
[IP Address]
   |
[Port 80]  [Port 443]  [Port 25]
   |          |           |
[Web]     [Secure Web] [Email]
```

<Axiom>An IP address says which machine; a port says which program on that machine. Neither one means anything without the other.</Axiom>

- **Fun Fact:** Port numbers are 16-bit values, so they range from 0 to 65535—65,536 possible doors per IP address, per transport protocol.

### Well-Known, Registered, and Ephemeral Ports

Not all port numbers are treated equally, and the ranges exist for good practical reasons:

| Range | Name | Typical Use |
|---|---|---|
| 0 - 1023 | Well-known ports | Standard services (HTTP 80, HTTPS 443, SSH 22); on Unix-like systems, binding to these usually requires admin/root privileges |
| 1024 - 49151 | Registered ports | Assigned to specific applications by IANA (e.g., many databases, game servers) but not requiring special privileges |
| 49152 - 65535 | Ephemeral (dynamic) ports | Temporarily assigned by the operating system to the *client* side of an outgoing connection |

<DeepDive title="Why your browser's port keeps changing">Every time your browser opens a new connection to a website, your operating system picks a fresh, essentially random port from the ephemeral range for the client side of that connection—this is exactly what you saw as the "local address" in the very first Go example in this part of the book. The server's port stays fixed and well-known (443 for HTTPS) so that anyone can find it, while the client's port only needs to be unique long enough to keep that one conversation's replies sorted from every other conversation your machine is having at the same time. This is also why you can open the same website in ten browser tabs simultaneously without them interfering with each other: each tab's connection gets its own distinct ephemeral port.</DeepDive>

<Warning title="A common misconception">"Privileged" ports below 1024 requiring root access is a security convention, not a law of physics baked into TCP/IP itself—it's an operating-system-level restriction, historically meant to reassure a client that a service answering on port 80 was set up by an administrator rather than an arbitrary unprivileged user. It says nothing about whether the service behind that port is actually trustworthy or well-written.</Warning>

## What is a Socket?

A socket is the combination of an IP address, a port number, and (implicitly) a transport protocol. It's the unique identifier that lets two programs on two machines talk directly, and in operating system terms, it's also the name for the actual programming handle—the file-descriptor-like object—your code uses to send and receive data once a connection exists.

- **Analogy:** If the IP is the building and the port is the room, the socket is the full address: "123 Main St, Room 80."
- **Example:**
  - `192.168.1.10:80` (IP:Port)

- **Diagram:**

```
[Client: 192.168.1.5:54321] <----> [Server: 203.0.113.10:80]
```

A single TCP connection is actually uniquely identified by *four* pieces of information at once—source IP, source port, destination IP, destination port—called a **socket pair** or **4-tuple**. This is precisely how a busy web server serving thousands of simultaneous clients keeps every single connection distinct, even though every one of them shares the exact same destination IP and port (its own).

- **Fun Fact:** Sockets are used for everything from web browsing to online gaming to the database connection behind almost every app you use.

---

## What is an Endpoint?

An endpoint is one end of a communication channel. It's defined by an IP address and a port, and sometimes a protocol (TCP/UDP)—in practice, "endpoint" and "socket address" are often used interchangeably, though "socket" can also refer to the live connection object itself, not just the address.

- **Analogy:** Like a phone number and extension—"Call 555-1234, ext. 80."
- **Example:**
  - TCP endpoint: `203.0.113.10:443`
  - UDP endpoint: `192.168.1.5:53`

- **Diagram:**

```
[Endpoint: 203.0.113.10:443/TCP]
```

- **Fun Fact:** Endpoints are how apps know exactly where to send and receive data—every function in Go's `net` package that opens a connection ultimately takes or returns one.

<DeepDive title="Why TCP and UDP ports don't collide">TCP port 53 and UDP port 53 are actually two completely independent "doors"—a program listening for TCP connections on port 53 and a different program listening for UDP packets on port 53 can coexist on the same machine without any conflict at all. This is exactly why DNS, which historically used UDP for ordinary lookups but falls back to TCP for larger responses (like zone transfers or big DNSSEC-signed answers), can run both on "port 53" simultaneously with no ambiguity—the transport protocol is part of the endpoint's true identity, even though we often casually drop it when we write `host:port`.</DeepDive>

## Real-World Example: Web Browsing

1. You type a website into your browser.
2. Your computer opens a socket to the server's IP and port 80 (or 443 for HTTPS), picking a random ephemeral port on its own side.
3. Data flows through this connection—your browser and the server are now talking through their respective endpoints, identified together by the full 4-tuple.

## Ports, Sockets, and Endpoints in Go

- **Go Example:**

```go
// Open a TCP connection to example.com on port 80
conn, err := net.Dial("tcp", "example.com:80")
if err != nil {
    log.Fatal(err)
}
defer conn.Close()
```

- **Explanation:** This code creates a socket (IP + port) and connects to the server's endpoint. Under the hood, `net.Dial` resolves `example.com` to an IP address, asks the operating system for an ephemeral local port, and performs the TCP handshake before returning a usable connection.

---

## Client vs. Server Roles at the Socket Level

| Aspect | Client socket | Server socket |
|---|---|---|
| Created with | `net.Dial` | `net.Listen`, then `Accept` |
| Port choice | Usually ephemeral, OS-assigned | Usually fixed, well-known |
| Who initiates | Client reaches out first | Server waits passively |
| Cardinality | One socket per outgoing connection | One listening socket accepts many incoming connections |

## Fun Facts & Notes
- Port 666 is sometimes nicknamed the "doom port," historically associated with the game Doom's networking, though it has no special protocol meaning.
- The highest possible port number is 65535, because ports are stored as 16-bit unsigned integers.
- A listening socket and a connected socket are different things even on the same port number: a web server's listening socket on port 80 spawns a brand-new connected socket for each client that connects, while continuing to listen for the next one.

### Try It Yourself in Go

Let's build both halves of a socket conversation in one program: a server that listens on a port and accepts a connection, and a client that dials it. Running them together makes the client/server, listen/dial, and local/remote address concepts from this chapter completely concrete.

```go
package main

import (
	"fmt"
	"net"
)

func main() {
	// Server side: listen on an OS-assigned free port (port 0 means
	// "pick any available port"), so this example is safe to run
	// repeatedly without a "port already in use" error.
	listener, err := net.Listen("tcp", "127.0.0.1:0")
	if err != nil {
		fmt.Println("Listen failed:", err)
		return
	}
	defer listener.Close()

	serverAddr := listener.Addr().String()
	fmt.Println("Server listening on:", serverAddr)

	// Accept exactly one connection in the background, and signal
	// main() when it's done printing so we don't exit early.
	done := make(chan struct{})
	go func() {
		defer close(done)
		conn, err := listener.Accept()
		if err != nil {
			fmt.Println("Accept failed:", err)
			return
		}
		defer conn.Close()
		fmt.Println("Server accepted connection from:", conn.RemoteAddr())
	}()

	// Client side: dial the server we just started.
	client, err := net.Dial("tcp", serverAddr)
	if err != nil {
		fmt.Println("Dial failed:", err)
		return
	}
	defer client.Close()

	fmt.Println("Client local socket:  ", client.LocalAddr())
	fmt.Println("Client remote socket: ", client.RemoteAddr())

	<-done // wait for the server goroutine to finish before exiting
}
```

Running `go run main.go` prints output similar to:

```
Server listening on: 127.0.0.1:61044
Client local socket:   127.0.0.1:61045
Client remote socket:  127.0.0.1:61044
Server accepted connection from: 127.0.0.1:61045
```

Look closely at the four numbers: the server's listening port (61044) is fixed and known in advance, exactly like a well-known port would be in a real service. The client's local port (61045) was picked by the operating system from the ephemeral range, exactly as described earlier in this chapter. And notice that "the client's remote socket" and "the server's listening address" are the same value, while "the server's accepted-connection remote address" and "the client's local socket" are also the same value—two matching pairs, from opposite points of view, which together form the one 4-tuple that uniquely identifies this single connection.
