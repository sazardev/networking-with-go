# Network Security Fundamentals

> "Imagine your network as a castle. You need walls, guards, secret codes, and clever traps to keep out intruders and protect your treasures!"

---

## Why Network Security Matters

Every day, billions of packets cross the internet — most are ordinary, but some are actively hostile: probing for open ports, guessing passwords, or trying to slip malicious code past a server that trusted its input too much. Network security is the discipline of assuming that hostile traffic exists and designing systems that stay safe anyway, rather than hoping attackers won't show up.

It's worth being specific about *why* this matters at internet scale, not just in the abstract. The 2017 WannaCry ransomware outbreak spread to over 200,000 computers across 150 countries in a matter of days by exploiting a single unpatched Windows SMB vulnerability — one missing security update, propagated automatically, worm-style, machine to machine. The 2016 Mirai botnet turned hundreds of thousands of poorly secured IoT devices (cameras, routers, DVRs, most still using factory-default passwords) into a weapon that knocked Twitter, Netflix, and Reddit offline by overwhelming a DNS provider called Dyn with traffic. Neither of these required a genius attacker — they required *one* weak link at scale. That's the core lesson of network security: it is rarely about stopping a brilliant adversary, and almost always about not being the weak link that gets automated attacks to succeed.

## The Pillars of Network Security

Security professionals organize almost everything around three goals, often called the **CIA triad** (nothing to do with the agency — just an unfortunate but memorable acronym):

### 1. Confidentiality
- **Analogy:** Whispering secrets so only the right people hear.
- **Goal:** Keep data private from anyone who isn't supposed to see it.
- **Mechanisms:** Encryption in transit (TLS, the subject of the deep dive below), encryption at rest (disk/database encryption), and access controls that limit who can even attempt to read the data.

### 2. Integrity
- **Analogy:** Sealing a letter with wax — if the seal is broken, you know someone tampered with it.
- **Goal:** Ensure data isn't altered, accidentally or maliciously, between sender and receiver.
- **Mechanisms:** Checksums catch accidental corruption; cryptographic hashes (SHA-256) and HMACs catch deliberate tampering; digital signatures additionally prove *who* produced the data.

### 3. Availability
- **Analogy:** Making sure the castle gates are always open for friends, but not for invaders.
- **Goal:** Keep services running and reachable for legitimate users.
- **Mechanisms:** Redundancy (multiple servers, multiple regions), rate limiting, and dedicated DDoS mitigation services that absorb or filter attack traffic before it reaches you.

<Warning title="Confidentiality without integrity is a false sense of security">It's tempting to think "it's encrypted, so it's safe," but encryption alone only protects confidentiality — it doesn't automatically prove the data wasn't tampered with or forged. This is exactly why TLS bundles encryption *with* message authentication codes and certificate-based identity checks. A system that encrypts data but never verifies its integrity can still be silently manipulated by an attacker who doesn't even need to read the plaintext, just flip bits in predictable ways. Confidentiality, integrity, and availability have to be considered together, not as a checklist where any one item alone is "enough."</Warning>

## Common Threats

- **Malware and ransomware:** Software designed to damage, spy on, or hold a system hostage. WannaCry (2017) encrypted victims' files and demanded payment, spreading automatically through an unpatched network vulnerability rather than needing anyone to click anything.
- **Phishing:** Tricking a person into handing over credentials or running malicious code, usually via a convincing fake email or website. It remains the single most common entry point for breaches precisely because it targets people, not infrastructure — no firewall stops an employee from typing their real password into a fake login page.
- **Man-in-the-middle (MITM) attacks:** An eavesdropper secretly intercepts, and possibly alters, traffic between two parties who believe they're talking directly to each other. A classic real-world version is ARP spoofing on open public Wi-Fi: an attacker on the same coffee-shop network tricks nearby devices into routing their traffic through the attacker's machine first.
- **Denial of Service / Distributed Denial of Service (DoS/DDoS):** Flooding a target with traffic or requests until it can't serve legitimate users. The 2016 Mirai botnet attack against DNS provider Dyn is the textbook example — hundreds of thousands of compromised IoT devices, acting together, generated enough traffic to take down major sites that depended on Dyn for DNS resolution.
- **Insider threats:** Sometimes the danger comes from someone who already has legitimate access — a disgruntled employee, a compromised contractor account, or simply a well-meaning employee who makes a costly mistake. This is why "trust but verify" access controls and audit logging matter even *inside* the perimeter, not just at it.

| Threat | How It Typically Works | Primary Defense |
|---|---|---|
| Malware / ransomware | Exploits unpatched software or tricks a user into running it | Patching, antivirus/EDR, backups |
| Phishing | Tricks a human into giving up credentials | User training, MFA, email filtering |
| Man-in-the-middle | Intercepts traffic on a shared or compromised network | TLS/encryption, certificate validation |
| DoS / DDoS | Overwhelms a target with traffic or requests | Rate limiting, redundancy, DDoS mitigation services |
| Insider threat | Misuses legitimate access | Least-privilege access, audit logging |

## Essential Security Tools & Techniques

### 1. Firewalls
- **Analogy:** The castle walls and gates — control who gets in and out.
- **Types:** Hardware appliances, host-based software firewalls, and cloud-native security groups. Chapter 11 goes into how firewalls actually decide what to block.

```
[Internet] ---[Firewall]--- [Internal Network]
```

### 2. Antivirus & Endpoint Detection
- **Analogy:** The castle's doctors — find and cure infections, and increasingly, watch for suspicious *behavior* rather than just known malware signatures.

### 3. Encryption
- **Analogy:** Sending secret messages in code that only the intended reader can decode.
- **Use Case:** HTTPS, VPNs, encrypted databases, encrypted backups.
- **Symmetric vs. asymmetric:** Symmetric encryption (like AES) uses one shared key for both encrypting and decrypting — fast, but both sides need the same secret key in advance. Asymmetric encryption (like RSA or elliptic-curve cryptography) uses a public/private key pair, so two strangers can establish trust without ever having shared a secret beforehand. In practice, TLS uses both: asymmetric cryptography to safely agree on a one-time shared key, then fast symmetric encryption for the actual data.

### 4. Authentication & Authorization
- **Analogy:** Checking IDs at the gate and only letting in the right people, for the right rooms.
- **Methods:** Passwords (ideally never stored in plain text, but hashed with a slow, salted algorithm like bcrypt or Argon2), biometrics, and two-factor/multi-factor authentication (2FA/MFA) — something you know plus something you have or something you are.

### 5. Intrusion Detection & Prevention Systems (IDS/IPS)
- **Analogy:** Security cameras and guards — spot and stop suspicious activity.
- **Signature-based vs. anomaly-based:** Signature-based systems recognize known attack patterns (fast, but blind to brand-new attacks); anomaly-based systems learn what "normal" traffic looks like and flag deviations (catches novel attacks, but with more false positives).

<DeepDive title="What actually happens during a TLS handshake">When your browser connects to an HTTPS site, it isn't just "turning on encryption" — it's running a small negotiation. The client and server first agree on a TLS version and a cipher suite. The server then presents its certificate, which the client validates against a chain of trust rooted in a small set of certificate authorities baked into the OS or browser — this is what proves the server is who it claims to be, not just that the connection happens to be encrypted. Once identity is established, both sides use asymmetric cryptography (in modern TLS 1.3, an ephemeral Diffie-Hellman key exchange) to agree on a shared symmetric key, without ever transmitting that key in the clear. From that point on, all application data — the actual HTTP request and response — is encrypted with that fast symmetric key. This is exactly what the Go example at the end of this chapter observes directly: the negotiated version, the cipher, and the certificate's subject and issuer.</DeepDive>

<Axiom>A lock only protects what's behind it if you also check who's holding the key — encryption without identity verification just means your secrets are safely delivered to whoever intercepted them.</Axiom>

## Security Best Practices

- **Keep software updated:** Patch vulnerabilities before attackers automate exploitation of them — WannaCry spread specifically because a patch had existed for months before the outbreak.
- **Use strong, unique passwords, plus MFA:** A leaked password alone shouldn't be enough to get in.
- **Educate users:** Phishing succeeds by targeting people, so people are part of the defense too.
- **Backup data, and test the restore:** A backup you've never restored from is a hope, not a plan.
- **Segment your network:** Don't let a compromised guest Wi-Fi device reach the same network as production servers.
- **Apply least privilege:** Every account and service should have only the access it actually needs — nothing "just in case."

## Real-World Example: Securing a Small Business

1. **Firewall:** Blocks unwanted inbound traffic from the internet.
2. **Antivirus/EDR:** Scans devices for malware and flags suspicious behavior.
3. **VPN:** Lets employees connect to internal resources securely from home or while traveling.
4. **MFA:** Adds a second factor beyond just a password for every login.
5. **Regular, tested backups:** Protects against ransomware and hardware failure alike.

## Try It Yourself in Go

Rather than just reading about TLS, you can watch a real handshake happen and inspect exactly what it negotiated — the protocol version, and the identity the server proved with its certificate.

```go
package main

import (
	"crypto/tls"
	"fmt"
	"net"
	"time"
)

func main() {
	host := "www.google.com"
	addr := net.JoinHostPort(host, "443")

	dialer := &net.Dialer{Timeout: 5 * time.Second}
	tlsConfig := &tls.Config{ServerName: host}
	conn, err := tls.DialWithDialer(dialer, "tcp", addr, tlsConfig)
	if err != nil {
		fmt.Println("TLS dial error:", err)
		return
	}
	defer conn.Close()

	state := conn.ConnectionState()
	fmt.Println("Negotiated TLS version:", tlsVersionName(state.Version))
	fmt.Println("Cipher suite:", tls.CipherSuiteName(state.CipherSuite))

	if len(state.PeerCertificates) > 0 {
		cert := state.PeerCertificates[0]
		fmt.Println("Certificate subject:", cert.Subject)
		fmt.Println("Certificate issuer:", cert.Issuer)
		fmt.Println("Valid until:", cert.NotAfter)
	}
}

func tlsVersionName(v uint16) string {
	switch v {
	case tls.VersionTLS10:
		return "TLS 1.0"
	case tls.VersionTLS11:
		return "TLS 1.1"
	case tls.VersionTLS12:
		return "TLS 1.2"
	case tls.VersionTLS13:
		return "TLS 1.3"
	default:
		return fmt.Sprintf("unknown (0x%04x)", v)
	}
}
```

Running this connects to a real, well-known HTTPS server and prints something like `Negotiated TLS version: TLS 1.3`, the cipher suite it chose, and the server certificate's subject and issuer (typically a large public certificate authority). `tls.DialWithDialer` performs the entire handshake described in the deep dive above before this program ever sees a byte of HTTP — certificate validation included, since `tls.Config` here has no `InsecureSkipVerify` set, meaning Go refuses to complete the connection at all if the certificate doesn't check out. This is the same validation your browser silently performs on every HTTPS page load, made visible instead of hidden behind a padlock icon.
