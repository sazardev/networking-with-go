# Network Troubleshooting and Tools

> "Imagine you're a digital detective, following clues through tangled wires and invisible signals to solve the mystery of a slow or broken network. With the right tools and a sharp mind, no problem is unsolvable!"

---

## Why Troubleshooting Matters

Networks are like busy highways — sometimes there's a traffic jam, a roadblock, or a detour. Troubleshooting is the art of finding and fixing these issues so data can flow smoothly again, and doing it *systematically* rather than by guessing. Whether you're a home user or a network engineer, knowing how to diagnose problems methodically is one of the most transferable skills in networking, because the same handful of tools apply whether you're debugging a home Wi-Fi issue or a production outage affecting thousands of users.

The most useful mental model for troubleshooting is to move **layer by layer**, from the bottom up (physical link, then IP reachability, then transport-level connectivity, then the application protocol itself) or top-down (does the app work? does the specific request work? does the raw connection work? does the network even reach the host?). Both directions are valid — bottom-up is more thorough, top-down is often faster when you have a hunch — but jumping randomly between layers without a plan is how a five-minute problem turns into an hour of confused guessing.

## Essential Troubleshooting Tools

### 1. Ping
- **Analogy:** Like shouting "Are you there?" and waiting for a reply.
- **Use Case:** Checks basic IP-layer reachability to a device, using ICMP echo request/reply messages.
- **Example:** `ping google.com`
- **What it doesn't tell you:** Ping only confirms that ICMP traffic can reach a host and come back — it says nothing about whether the *service* you actually care about (a web server, a database) is running or reachable, since ICMP and, say, TCP port 443 are handled completely independently by firewalls.
- **Fun Fact:** The name comes from sonar pings used by submarines to detect nearby objects by listening for an echo.

### 2. Traceroute (`tracert` on Windows)
- **Analogy:** Like following a package's journey through every post office it visits.
- **Use Case:** Shows the path data takes to reach its destination, hop by hop, and where along that path delay or loss appears.
- **Example:** `tracert example.com` (Windows) or `traceroute example.com` (Linux/Mac)
- **How it actually works:** Traceroute sends a series of packets with a deliberately small **TTL** (time-to-live), starting at 1. Each router along the path decrements the TTL by one, and a router that decrements it to zero discards the packet and sends back an ICMP "time exceeded" message — revealing itself as the hop at that distance. Traceroute repeats this with TTL 2, then 3, and so on, building the full path one hop at a time purely by watching who complains.

### 3. `ipconfig` / `ifconfig` (or `ip addr` on modern Linux)
- **Analogy:** Like checking your own address and ID.
- **Use Case:** Displays your device's network settings — assigned IP address, subnet mask, default gateway, DNS servers.
- **Example:** `ipconfig` (Windows), `ifconfig` (older Linux/Mac), `ip addr` (modern Linux)

### 4. `nslookup` / `dig`
- **Analogy:** Looking up a friend's phone number in a directory.
- **Use Case:** Queries DNS directly, helping diagnose name-resolution issues separately from everything else that could be wrong.
- **Example:** `nslookup github.com`, `dig github.com`, or `dig github.com MX` to check a specific record type.

### 5. `netstat` (or `ss` on modern Linux)
- **Analogy:** Like checking all the open doors and windows in your house.
- **Use Case:** Lists active network connections and listening ports, along with each connection's state — `LISTEN` (waiting for connections), `ESTABLISHED` (an active, working connection), `TIME_WAIT` (a connection that recently closed and is lingering to catch any stray late packets).
- **Example:** `netstat -an`

### 6. Wireshark
- **Analogy:** Like a microscope for network traffic — see every packet that flows by, byte for byte.
- **Use Case:** Deep packet-level analysis for hard problems (protocol-level bugs, security investigations, working out exactly which side dropped a connection and why).
- **Fun Fact:** Wireshark can decode hundreds of protocols out of the box, from HTTP down to obscure industrial control system protocols.

### 7. `curl` / manual protocol probing
- **Analogy:** Speaking the protocol yourself instead of asking a translator to do it for you.
- **Use Case:** `curl -v https://example.com` shows the DNS lookup, TCP connect, TLS handshake, and HTTP exchange all in one command with verbose timing — extremely useful for narrowing down *which* layer of a "the website is broken" complaint is actually failing.

<DeepDive title="Reading traceroute output usefully">A traceroute that shows increasing latency evenly across many hops usually just reflects real geographic distance — nothing is wrong. What's actually diagnostic is a *sudden jump*: if hop 8 responds in 20ms and hop 9 suddenly takes 400ms, and every hop after that is also around 400ms, you've likely found where the delay is being introduced. Be careful about one common trap, though: some routers deprioritize or rate-limit the ICMP replies traceroute depends on (because generating those replies isn't their main job), so a hop that shows `* * *` (no reply) isn't necessarily broken — it might just be a router that doesn't bother answering, while everything past it still works fine.</DeepDive>

<Warning title="A successful ping does not mean the service you care about is reachable">This is one of the most common troubleshooting mistakes: `ping` succeeding is often treated as proof that "the server is up." Ping only tests ICMP reachability at the IP layer. Many servers and firewalls deliberately allow ICMP while blocking or rejecting a specific TCP port, or vice versa — a server can happily answer pings while its web server process has crashed, and it can just as easily block ICMP entirely (a common hardening practice) while its actual services work perfectly. If you need to know whether a *service* is reachable, test the actual port, e.g. with `curl`, `telnet host port`, or a small program like the one at the end of this chapter — don't stop at ping.</Warning>

## Troubleshooting Workflow

1. **Identify the problem precisely:** "The internet is down" is not a diagnosis — is it one site, one device, one app, or everything?
2. **Gather information layer by layer:** Physical link up? IP address assigned (`ipconfig`/`ifconfig`)? Can you reach the default gateway? Can you resolve DNS (`nslookup`)? Can you reach the destination IP (`ping`)? Can you reach the specific port (`curl`, a manual dial)?
3. **Isolate the issue:** Is it your device, your local network, your ISP, or the remote server? Testing from a different device or network is one of the fastest ways to narrow this down.
4. **Test a hypothesis, one change at a time:** Restart a device, check a cable, adjust a firewall rule — but change one variable before retesting, or you won't know which change actually fixed it.
5. **Verify the fix, and understand why it worked:** Confirming the symptom is gone is good; understanding the root cause is what prevents the same issue next month.

## Tool-to-Layer Cheat Sheet

| Tool | Primarily Checks | Rough OSI Layer |
|---|---|---|
| `ipconfig` / `ifconfig` | Local interface configuration | Layer 3 (and below) |
| `ping` | Basic IP reachability | Layer 3 |
| `traceroute` | Path and per-hop latency | Layer 3 |
| `nslookup` / `dig` | Name resolution | Layer 7 (application, DNS) |
| `netstat` / `ss` | Local connection/port state | Layer 4 |
| `curl` / manual dial | End-to-end application behavior | Layer 4 through 7 |
| Wireshark | Everything, packet by packet | All layers |

<Axiom>Troubleshooting is the discipline of turning "it's broken" into a specific, testable claim about one layer at a time — the tool matters less than the order you reach for it.</Axiom>

## Try It Yourself in Go

Command-line tools like `ping` and `curl` are great, but you can build a purpose-made connectivity checker in a few lines of Go — genuinely useful for scripting health checks against a known list of dependencies.

```go
package main

import (
	"fmt"
	"net"
	"time"
)

type target struct {
	name string
	addr string
}

func main() {
	targets := []target{
		{"Google DNS", "8.8.8.8:53"},
		{"Cloudflare DNS", "1.1.1.1:53"},
		{"Example over HTTP", "example.com:80"},
		{"Likely closed port", "127.0.0.1:9"},
	}

	for _, t := range targets {
		checkTarget(t)
	}
}

func checkTarget(t target) {
	start := time.Now()
	conn, err := net.DialTimeout("tcp", t.addr, 3*time.Second)
	elapsed := time.Since(start)

	if err != nil {
		fmt.Printf("%-20s %-20s DOWN  (%v)  %v\n",
			t.name, t.addr, elapsed, err)
		return
	}
	defer conn.Close()
	fmt.Printf("%-20s %-20s UP    (%v)\n", t.name, t.addr, elapsed)
}
```

Running this prints one line per target, showing whether the TCP handshake to that `host:port` succeeded, how long it took, and the underlying error when it failed (connection refused, timeout, no route to host, and so on all look different and each hints at a different layer of failure — exactly like the workflow above). This is deliberately closer to `curl -v` than to `ping`: it tests actual TCP reachability at a specific port, which, as the warning above explains, is a meaningfully different and often more useful question than "does this host respond to ICMP."

## Fun Facts & Memes
- The first network bug was literally a moth stuck in a computer relay, in 1947.
- "Have you tried turning it off and on again?" remains one of the most effective fixes in tech history, because it resets accumulated state that a layer-by-layer check might not catch quickly.
- Wireshark's shark-fin logo is a nod to its predecessor tool, Ethereal — its mission was, and still is, to make invisible packets visible.
