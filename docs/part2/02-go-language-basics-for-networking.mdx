# Go Language Basics for Networking

> "Learning Go for networking is like getting a Swiss Army knife for the digital world—simple, sharp, and ready for anything!"

---

## Why Go for Networking?

Go (Golang) was born at Google to solve real-world problems: speed, simplicity, and scalability. Its networking capabilities are legendary—think of Go as the friendly engineer who makes building servers, clients, and protocols a breeze.

- **Analogy:** If C is a race car and Python is a comfy sedan, Go is a Tesla—fast, modern, and easy to drive.
- **Fun Fact:** Go’s mascot, the gopher, is so iconic that it has its own plushies, comics, and even a song!

This chapter isn't a full Go tutorial — it's the subset of the language
you'll lean on constantly once you start writing servers and clients: how
to model a connection with a struct, read bytes into a slice, report a
dropped connection with an error, and serve many clients at once with
goroutines. Later chapters assume you're comfortable with everything below.

<Axiom>Go's networking code isn't a special dialect of the language — it's ordinary Go: structs, errors, and goroutines, applied to bytes moving over a wire.</Axiom>

## Go Syntax: The Essentials

Let’s review the Go basics you’ll use in every networking project.

### 1. Hello, Go!

```go
package main
import "fmt"
func main() {
    fmt.Println("Hello, Go Networking!")
}
```

[Try this exercise ](../../exercises/part2/02-hello-go/main.go)

<DeepDive title="package main and func main">Every Go executable needs exactly one `package main` with exactly one `func main()` — the entry point the linker wires up. `go run` compiles to a temp binary and runs it in one step; `go build` produces a standalone binary you can copy anywhere, which matters once you're deploying a server.</DeepDive>

<Warning title="go run vs go build for anything long-running">`go run` is great for quick checks, but it recompiles on every invocation. For a server you'll actually deploy, `go build -o server` once and run the resulting binary instead.</Warning>

### 2. Variables and Types

Go is statically typed, but super friendly:

```go
var message string = "Networking is fun!"
port := 8080 // Short declaration
fmt.Printf("%s Listening on port %d\n", message, port)
```

[Exercise: Variables and Types](../../exercises/part2/02-variables-and-types/main.go)

<DeepDive title="Zero values and type inference">Go never leaves a declared variable undefined — `var port int` starts at `0`, `var host string` at `""`. The `:=` short form infers the type from the right side, so `port := 8080` is an `int` — but a real port number is never negative or above 65535, and the type alone doesn't enforce that.</DeepDive>

<Warning title="int is not a port number">A TCP/UDP port is a 16-bit unsigned value (0-65535), but Go code almost always stores it in a plain `int`. Nothing stops you from writing `port := 99999` and only discovering the mistake when `Dial` returns a cryptic error at runtime — validate user-supplied ports early.</Warning>

### 3. Functions

Functions are first-class citizens in Go:

```go
func greet(name string) string {
    return "Welcome, " + name + "!"
}
fmt.Println(greet("Gopher"))
```

[Exercise: Functions](../../exercises/part2/02-functions/main.go)

<DeepDive title="Multiple return values, everywhere">Almost every networking call in Go returns two values: a result and an error, like `conn, err := net.Dial(...)`. Ordinary functions like `greet` above could just as easily return `(string, error)` — once that pattern feels natural, `net`, `io`, and `bufio` signatures stop feeling foreign.</DeepDive>

<Warning title="Discarding one of two return values">`conn, _ := net.Dial("tcp", addr)` is syntactically valid but throws away the error — betting the connection always succeeds, a bet that fails the first time DNS is slow or the remote host is down. Name and check every `err` you receive.</Warning>

### 4. Control Structures

Go keeps it simple:

```go
for i := 1; i <= 3; i++ {
    fmt.Println("Packet", i)
}
if port == 8080 {
    fmt.Println("Standard HTTP port!")
}
```

[Exercise: Control Structures](../../exercises/part2/02-control-structures/main.go)

<DeepDive title="for is Go's only loop keyword">Go has no `while` or `do-while` — `for` covers every case. The three-clause form is above; `for condition { }` behaves like `while`; bare `for { }` loops forever, exactly the shape of an accept loop waiting for the next client. `range` is a special form of `for` for slices, maps, and channels.</DeepDive>

<Warning title="A bare for {} needs its own exit">`for { }` never stops on its own — it relies on `break`, `return`, or the body blocking (like `ln.Accept()` waiting on the network) to avoid spinning your CPU at 100%. A bare `for {}` around non-blocking work with no exit condition is a busy-loop, not a server.</Warning>

### 5. Structs and Methods

Go’s structs are like blueprints for data:

```go
type Server struct {
    Host string
    Port int
}
func (s Server) Address() string {
    return fmt.Sprintf("%s:%d", s.Host, s.Port)
}
```

`Address` uses a **value receiver** — `s` is a copy of the `Server` it was
called on, which is fine here since we only read fields. Once a struct
starts holding a live connection or a mutex, that changes:

```go
type Client struct {
    conn net.Conn
    tag  string
}

func (c *Client) Connect(addr string) error {
    conn, err := net.Dial("tcp", addr)
    if err != nil {
        return err
    }
    c.conn = conn
    return nil
}
```

<DeepDive title="Value receivers vs pointer receivers">`Connect` uses a **pointer receiver** (`*Client`) on purpose: it mutates `c.conn`, and a value receiver would only set the field on a throwaway copy, doing nothing to the original. Rule of thumb: if a method changes the struct, or the struct holds a `net.Conn`, `sync.Mutex`, or buffer that shouldn't be copied, give it a pointer receiver.</DeepDive>

<Warning title="Copying a struct that holds a connection">Passing a `Client` by value copies the struct, `net.Conn` field included. Both copies now share the same underlying connection, but writes to one copy's other fields never reach the original — a frequent source of "why didn't my change take effect?" bugs. Pass connection-holding structs by pointer.</Warning>

### 6. Slices, Maps, and Ranges

Go’s slices and maps are perfect for handling connections and routing tables:

```go
connections := []string{"client1", "client2"}
for _, c := range connections {
    fmt.Println("Connected:", c)
}
ports := map[string]int{"http": 80, "https": 443}
for name, port := range ports {
    fmt.Printf("%s => %d\n", name, port)
}
```

[Exercise: Slices and Maps](../../exercises/part2/02-slices-and-maps/main.go)

<DeepDive title="Slices are how you'll move bytes over the network">A slice is a small header (pointer, length, capacity) over an underlying array. `make([]byte, 4096)` allocates a buffer once; `conn.Read(buf)` fills part of it and returns how many bytes it wrote, so you slice `buf[:n]` rather than assuming the whole buffer was used — the pattern behind every `Read`/`Write` call from Chapter 5 onward.</DeepDive>

<Warning title="Map iteration order is not deterministic">Go deliberately randomizes the order `range` visits a map's keys, on every run — the `ports` loop above prints in a different order each time. If you're building a routing table where check order matters, sort the keys explicitly rather than relying on map iteration order.</Warning>

### 7. Error Handling

Go’s error handling is explicit and honest:

```go
conn, err := net.Dial("tcp", "localhost:8080")
if err != nil {
    log.Fatal(err)
}
```

<Warning title="log.Fatal skips your defers">`log.Fatal` logs and then calls `os.Exit(1)` immediately, so any `defer conn.Close()` written earlier never runs. Harmless in a one-shot script, but inside a long-running server, calling `log.Fatal` on a per- connection error kills the *entire process* over one bad client. Prefer `log.Println` plus `return` (or `continue` in an accept loop) instead.</Warning>

<DeepDive title="Checking *what kind* of error you got">A plain `if err != nil` only tells you *something* went wrong. `net` package errors implement `net.Error`, which adds `Timeout() bool`, so `if ne, ok := err.(net.Error); ok && ne.Timeout()` can retry on a timeout but give up on a permanent failure. Wrapped errors (`%w`) still match correctly through `errors.Is(err, io.EOF)`.</DeepDive>

### 8. Goroutines and Channels (Concurrency Magic)

Go’s secret sauce for networking is easy concurrency:

```go
go func() {
    fmt.Println("Handling connection in a goroutine!")
}()
```

Channels let goroutines talk:

```go
messages := make(chan string)
go func() { messages <- "Ping!" }()
fmt.Println(<-messages)
```

<DeepDive title="Unbuffered vs buffered channels">`make(chan string)` above is **unbuffered**: the send blocks until something receives, and the receive blocks until something sends. That handshake is what makes the two-line example work without a `sync.WaitGroup`. `make(chan string, 10)` instead creates a **buffered** channel that holds up to 10 values before a send blocks — useful for smoothing out bursts.</DeepDive>

<Warning title="A goroutine per connection isn't free">Spawning `go handleConn(conn)` per connection is the standard Go pattern, and goroutines are cheap (a few KB of stack each) — but not free. A server under a connection flood with no limit on concurrent goroutines can still exhaust memory or file descriptors; real servers eventually pair this pattern with a connection limit or a worker pool.</Warning>

<Warning title="Sending with no receiver deadlocks">Writing `messages <- "Ping!"` on the same goroutine that will later read `<-messages`, with no `go` in front of the send, deadlocks: the send blocks forever waiting for a receiver that can never run. Go's runtime detects this and panics with "all goroutines are asleep — deadlock!" rather than hanging silently.</Warning>

### 9. Goroutines Meet the Network: The Accept Loop

Every goroutine and channel concept from Go Fundamentals, Chapter 9,
exists in this book for one killer use case: the accept loop. A
`net.Listener`'s `Accept()` blocks until a client connects, hands back a
`net.Conn`, and then blocks again waiting for the next one — spawning a
goroutine per connection lets the loop go straight back to blocking
instead of waiting for the current client to finish first:

```go
ln, err := net.Listen("tcp", ":9000")
if err != nil {
    log.Fatal(err)
}
for {
    conn, err := ln.Accept()
    if err != nil {
        log.Println("accept error:", err)
        continue
    }
    go handleConn(conn) // one goroutine per client; loop keeps spinning
}
```

Every TCP and UDP server for the rest of this book is a variation on
this same shape: block on `Accept`, spawn, repeat. A goroutine costs a
few KB of stack rather than a whole OS thread, so `go handleConn(conn)`
scales to thousands of simultaneous connections with no extra ceremony
— it's one of the main reasons Go is popular for network services.

It doesn't scale to *infinity*. Ten thousand slow, mostly-idle clients
are cheap; a flood of a million connections in the same few seconds
still means a million live goroutines, each holding a stack, a
`net.Conn`, and whatever buffers the handler allocates — memory and
file descriptors run out even though goroutines don't. The escape
hatch, covered in full in Chapter 8 of Part 2 (Concurrency in
Networking), is a **worker pool**: a fixed number of goroutines pulling
connections off a channel instead of one goroutine per connection.
Treat the loop above as the default, and reach for a worker pool only
once you've actually measured that unbounded goroutines are the
problem.

<DeepDive title="Why the accept loop doesn't need a mutex on conn">Each trip through `for { conn, err := ln.Accept(); ... }` declares a brand-new `conn` with `:=`, so unlike a variable declared once outside the loop, nothing is shared between the goroutines spawned by different iterations — each closes over its own private connection. Synchronization only becomes necessary once handlers start touching something *outside* their own connection, like a shared map, log file, or counter.</DeepDive>

<Warning title="The loop variable trap, before and after Go 1.22">Go 1.21 and earlier reused a single loop variable across every iteration of a `for range`, so `for _, addr := range addrs { go func() { dial(addr) }() }` could have every goroutine dial whatever `addr` happened to be *last* by the time they actually ran — the classic fix was passing it in explicitly, `go func(a string) { dial(a) }(addr)`. Go 1.22 changed the language so each iteration gets its own fresh copy of the variable, fixing this whole class of bug by default — but plenty of code (and this book) still passes the variable in explicitly, since it stays correct on any Go version and makes the capture obvious to the reader.</Warning>

<Warning title="A shared net.Conn still needs its own synchronization">The standard library documents `net.Conn` as safe for multiple goroutines to call methods on simultaneously, so nothing panics if one goroutine reads while another writes on the same connection — but "safe" only means no data race, not "protocol-correct." Two goroutines calling `Write` on the same connection at the same time can have their bytes interleave on the wire, corrupting whatever framing your protocol expects. A connection written to from more than one goroutine still needs a mutex, or a single writer goroutine fed by a channel.</Warning>

## Real-World Example: Simple TCP Client

```go
package main
import (
    "fmt"
    "net"
    "os"
)
func main() {
    conn, err := net.Dial("tcp", "example.com:80")
    if err != nil {
        fmt.Println("Error:", err)
        os.Exit(1)
    }
    fmt.Fprintf(conn, "GET / HTTP/1.0\r\n\r\n")
    buf := make([]byte, 4096)
    n, _ := conn.Read(buf)
    fmt.Println(string(buf[:n]))
    conn.Close()
}
```

<Warning title="conn.Close() at the end of main isn't a safe habit">`conn.Close()` runs at the bottom of `main` above — but if `conn.Read` panicked, or a future edit added an early `return`, the connection would leak: the file descriptor stays open until the process exits. `defer conn.Close()` right after `Dial` guarantees it always runs.</Warning>

<Warning title="The ignored Read error, and partial reads">`n, _ := conn.Read(buf)` also discards its error, and reads only *once*. TCP has no concept of "messages" — a single `Read` may return part of a response, all of it, or signal `io.EOF`. One `Read` call is not guaranteed to capture a whole HTTP response, especially past a few KB.</Warning>

A more careful version fixes both by deferring the close and looping over
`Read` until `io.EOF`:

```go
package main

import (
	"fmt"
	"io"
	"net"
	"os"
)

func main() {
	conn, err := net.Dial("tcp", "example.com:80")
	if err != nil {
		fmt.Println("Error:", err)
		os.Exit(1)
	}
	defer conn.Close()

	fmt.Fprintf(conn, "GET / HTTP/1.0\r\n\r\n")

	buf := make([]byte, 4096)
	for {
		n, err := conn.Read(buf)
		fmt.Print(string(buf[:n]))
		if err == io.EOF {
			break
		}
		if err != nil {
			fmt.Println("read error:", err)
			break
		}
	}
}
```

Note `buf[:n]` prints *before* checking `err` — `Read` can return both
data and `io.EOF` in the same call, so discarding it on error would drop
the tail of the response.

### Try It Yourself: Count the Bytes

Extend the careful client above to report how much data it received:

1. Add a `total := 0` before the loop, and `total += n` each iteration.
2. After the loop ends, print `fmt.Println("bytes received:", total)`.
3. Change the request line to `HEAD / HTTP/1.0\r\n\r\n` and compare the
   byte count — a `HEAD` response has headers but no body.
4. **Bonus:** wrap the `Dial` call in a loop that retries up to 3 times
   with a short pause, printing which attempt finally succeeded.

<Axiom>Every byte a TCP connection hands you comes through the same three-step ritual: read into a slice, check the error, decide whether to keep going.</Axiom>

## Frequently Asked Questions

**Should Address() on my Server struct use a value or pointer receiver?**
It depends on whether the method reads or mutates the struct. `Address` in this chapter only reads `Host` and `Port`, so a value receiver is fine—but `Connect` on the `Client` struct assigns to `c.conn`, and a value receiver there would only update a throwaway copy, leaving the original struct's `conn` field untouched. The rule of thumb: if a method changes the struct, or the struct holds a `net.Conn`, mutex, or buffer, give it a pointer receiver.

**Why does my map of ports print in a different order every time I run the program?**
That's Go being deliberate, not a bug. The runtime randomizes the order `range` visits a map's keys on every single run, specifically so nobody accidentally depends on an order that was never guaranteed. If you're building something like a routing table where check order matters, sort the keys explicitly instead of trusting iteration order.

**My program printed "all goroutines are asleep - deadlock!" What did I do wrong?**
You almost certainly sent on an unbuffered channel with nothing ready to receive it. `messages <- "Ping!"` on the same goroutine that will later run `<-messages`, with no `go` in front of the send, blocks forever waiting for a receiver that can never start—Go's runtime detects that the whole program is stuck and panics rather than hanging silently forever.

**If goroutines are so cheap, why does the chapter warn about spawning one per connection?**
Because "cheap" isn't "free." A few KB of stack per goroutine handles thousands of slow, mostly-idle clients without breaking a sweat, but a flood of a million connections in the same few seconds still means a million live goroutines, each with its own stack, `net.Conn`, and handler buffers—memory and file descriptors run out even though the goroutines themselves stay lightweight. That's exactly why Part 2's concurrency chapter later introduces worker pools as the escape hatch, reached for only once you've measured that unbounded goroutines are the actual problem.

**Why does the careful TCP client print `buf[:n]` before checking whether `err` is `io.EOF`?**
Because `Read` is allowed to return both data and `io.EOF` in the same call—the last chunk of the response can arrive at the exact moment the connection closes. Checking the error first and discarding the data on `io.EOF` would silently drop the tail end of whatever you were reading, which is why the loop prints first and only then decides whether to break.
